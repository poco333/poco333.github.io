期間

Baitryは、「バイトを体験してみたい方」と「バイトを募集しているお店」を結ぶ体験バイトに特化したマッチングサービスです。

人材不足が社会課題となっている中、一つひとつの出会いのミスマッチをなくし、より良い出会い方のスタンダードになる未来を目指しています。

今回、ワーカー、勤怠管理専用のショップ側共に「1.新しくなったコンセプト（世界観）を元にビジュアルをリニューアルする」こと、そして「2.ビジュアルのリニューアルと同時にUI改善」を行いまいした。

Introduction

リニューアル背景

以前はスポットメイトという、「空いた時間（スポット）でバイトができる」の意味合いが強いコンセプトでしたが、今後は、空いた時間（スポット）でただ単にバイトするのではなく、空いた時間を「活用」して「お試しバイトができる」といった方針への変更が挙げられていました。

そこで、改めてネーミングやコンセプトの見直し、そしてコンセプトに紐付いたビジュアルのリニューアル及びUI改善を行っていく必要がありました。

ミッション

リニューアル背景を元にiOSを要とし、Android、Webへ展開していく流れで進められました。

私は、iOSアプリを担当することになったため、「1.新しくなったコンセプト（世界観）からビジュアルへの落とし込み(リニューアル)」「2.ビジュアルへの落とし込みと同時にUI改善も行う」に取り組みました。

  1. 新しくなったコンセプト（世界観）をビジュアルに落とし込む
  2. ビジュアルのリニューアルと同時にUI改善も行う

以下、ワーカー側のみ新コンセプトに沿ったイラスト及びカラーが既に画面へ適用されていたため更にコンセプト（世界観）を反映できるよう方針をかためていきました。

Concept

コンセプトの把握

まずは、決定する前に新ロゴに込められたコンセプトの把握を行いました。

Let`s try it together

楽しくて自分にあったバイトを紹介してくれる存在。それがBaitryです。 求人なんて見てもよくわからない..それなら気になるところですぐに働こう。 そんな気軽で楽しいお試しバイトはじまります。

ビジュアルの目指す方向

そして、コンセプトの方針を基に、どのようなビジュアルで進めるのかを以下に定めました。

・全体として、単なるバイト探しではなく、「出会いの後押し」として、「気軽で楽しくハッピー」な雰囲気作りを目指す

・ロゴのように元気さと楽しさ、気軽さを合わせた雰囲気を踏襲する

Visualize

方法1：カラーの統一

以前は「イエロー ＋ ブルー」の2色で展開されており、ブルーは主にネガティブな印象に使用されていましたが、 今回の、ハッピーな出会いの後押しという意味からもずれてしまう点、そしてメインのイエローよりもブルーが目立ってしまうことからも考えてブルーの使用をやめ、「プライマリーカラーをイエロー」のみとしました。

方法2：デザイン手法の統一

バイトリーでは「応援する存在」として人感や情緒感を演出し出していきたいと考えていました。 昨今のトレンドとして、App StoreやiTuens、Mac公式サイトなどでもよく見受けられる、 モダンでコンテンツ重視な設計は、バイトリーの「応援する存在」として没入感や情緒感を演出しやすいのでは？と考えました。 また、20代の学生がターゲットという点も含めて今回のプロダクトに反映することにしました。

方法3：イラストの活用

以前は、重要なコミュニケーションのタッチポイントに上手く活用しきれていない感がありました。

そのためイラストを使用する意味を考え、どう活用するのが効果的かを定義し、以下に合わせイラストの見せ方を変えていきました。

活用1：例えば、印象を和らげる時

応募する際に無断欠勤や選考辞退を防ぐためのモーダル画面にて、以前のただ文面を並べた注意する画面ではなく、 応募意欲が高いことも踏まえてお店の人との約束事としてポジティブに伝える見せ方をしました。

活用2：例えば、メッセージをユーザーに伝える時

アプリでは写真登録を必須にできないため、写真登録促進をイラストで魅力的に伝えたり、応募完了や、アップデート画面、お知らせ促進画面など一画面活用して伝えたいメッセージがある時などに イラストを一番多く活用しました。

また、学生の募集管理画面で機械的に募集案件が並ぶ画面にて吹き出しを入れ、応援する存在としてイラストを活用しました。

Output

UI Change

応募から勤務までスムーズに操作できるよう、UI改善を行いました。以下、改善した一部を紹介します。 前提として、大枠の設計（流れ）への変更ではなく、今回のリニューアルの目的は新しくなった世界観をUIに反映し、それに応じて改善を行うことでした。（その後に大枠も含めて改善を行っていきます） スタートアップのリリース間もない時期でもあり、スピード重視であることから、まずは「各画面の目的に対してUIがしっかり機能しているか」に観点をおいて改善を進めました。

募集一覧と募集詳細（サービスのメインとなる画面）

前提

UI変更をするにあたり、まずは前提情報として、画面の目的及び理想となるユーザー体験を定義しました。

目的
募集一覧：ユーザーが空いた時間を活用して行う、体験バイト探し
詳細：体験内容の詳細や状況確認、そして応募、辞退、連絡をする

タッチポイント
起動直後、検索時や検討時

制約
・現段階ではUI変更のみであるため（コストがかかる）、大枠の設計（流れ）は変えられない。
・詳細に検索できるほどの体験バイトの募集はまだない。

理想状態

ユーザーが望んでいる空き日時に体験したいバイトに出会い、応募できること。 その際に、①どこで②何を③どのようにするのか、がスムーズに分かること。

現状（課題）

前提を踏まえ、現状はどのような状態になっているのか、理想状態を目指すためにどこが課題となりそうかを整理しました。

全体

以前がスポット（空き時間）を重視したコンセプトであったため、そのまま「いつ」に重点が置かれたUIになっている。

詳細

・日にちが一番最初に目に入り、次に勤務時間が目に入りやすい。そして、応募の制限時間もリアルタイムで更新されているため時間要素が強くなっている。

・「何をする（役割、職務）」のかに関する情報はそれほど重視されていない。

・自身の空き日時を探す際には、上下にスクロールしなくてはならず、ユーザーにとって負荷がかかる仕様になっている。

解決
上記などを踏まえ、あるべきUIの姿を定義し、どのように実現できるかを整理しました。

全体

スポット（空き時間）重視ではなく、空き時間を活用してできる「体験バイト=コンテンツ」重視なUIであること。

また、最終的には体験バイトを通して長期や短期の雇用に繋げることを目指しているため、人や雰囲気、情緒感が伝わりやすいUIであること。

詳細

・「何をする」に重点を置けるようため、どんな求人なのかを表す「タイトル」をメインとして配置。

・応募締め切りまでの時間に関してはリアルタイムで更新されるのではなく、残り〇〇日や〇〇時間として表示させ、時間に急かされる感覚を緩和。

・日にちの一覧姓を良くするために上下スクロールではなく、カレンダー表示として横にスライドさせる仕様に変更。

・全体のテキストをBoldにすることで、コンテンツや募集としての力強さを表現。

その他

店舗の基本情報を「①日時②駅③住所④時給」から「①駅②住所③時給④日時」に変更しました。上記に変更した理由は以下です。

① スポット（いつ）からコンテンツ（何を）重視に変更のため、日時の優先順位が下がった。

② 募集一覧画面では「①いつ②何をする③どれくらいの時給で」がメインで分かるようになっていたため、詳細画面では「④どこで」を重視して「場所」に関する情報を 一番上にし、日時と時給をその次に配置。

応募管理

前提

UI変更をするにあたり、まずは前提情報として、画面の目的及び理想となるユーザー体験を定義しました。

目的
応募管理：応募したアルバイト先の状況確認

タッチポイント
応募状況の確認時（選考中、採用済みなど）、採用や不採用後の確認時

制約
現段階ではUI変更のみであるため（コストがかかる）、大枠の設計（流れ）は変えられない。

理想のユーザー体験

ユーザーが応募後に今どんな状況かを確認でき、そしてどういった結果になったのかを履歴として知れること。

現状（課題）

前提を踏まえ、現状はどのような状態になっているのか、理想状態を目指すためにどこが課題となりそうかを整理しました。

全体

応募後の処理がすべて一画面に集約されて積み上げ状態になっており、情報構造としても扱いづらい。上下スクロールではない形でプロセスに沿ってカテゴリ分けする必要がある。

詳細

採用、不採用などを表示するために透過レイヤーがかかっているが、肝心の店舗情報が見えなくなっている。

解決

現状からどのように分けて管理されるべきかを探るため、ユーザーフローを通して情報の可視化を行いました。

ユーザーフロー

勤務の場合
応募する → 応募管理にて応募した一覧を確認（選考中） → 採用通知 → 応募管理にて採用済みを確認 → 勤務 → 勤務後、勤務したかどうかの確認通知 → 勤務済みを選択

選考辞退の場合
応募する → 応募管理にて応募した一覧を確認（選考中） → 選考辞退

お見送りの場合
応募する → 応募管理にて応募した一覧を確認（選考中） → 不採用通知

欠勤の場合
応募する → 応募管理にて応募した一覧を確認（選考中） → 採用通知 → 応募管理にて採用済みを確認 → 勤務後、勤務したかどうかの確認通知 → 欠勤を選択

上記の可視化から「勤務に至るまでの状況確認」と「選考や勤務の終了後に履歴として積み上がる」パターンとで区切って表示していく仕様が良さそうでした。 そのため、「勤務に至るまでの状況確認」をメインとし、終了後は履歴に格納させる形で別々で作成することとしました。

実際のUIに落とし込む際は状況確認をメインとし、「選考中」と「採用済み」をセグメンテッドコントロールで分けました。 履歴は別画面を新しく用意し、こちらでも「勤務済み」と「見送り」で分けて設計しました。

学生向けプロフィール設定

前提

最初にプロフィール設定画面の目的や前提を整理しました。

目的
店舗が選考する際に確認するための情報（写真、氏名、生年月日など）。

タッチポイント
登録直後、応募時、マイページ → コストをいずれの画面にも同じUIを使用。

制約
・現段階ではUI変更のみであるため（コストがかかる）、大枠の設計（流れ）は変えられない。
・写真登録は重要だがAppStoreの規約上、登録必須にできない。
・姓と名の間にスペースを入れて入力してもらわないといけない

あるべきUI

まずは機能として「必ず登録しなければならない情報が分かり、ストレスなく登録できる」を満たしている事が必要

課題

全体

・必須項目が分かりづらく、必ず登録しなければならない情報かどうかの見分けがつきにくい。 （→ 写真面積が大きいため、タイトルの「必須項目」がフォームの範囲まで含まれているのかどうか分かりづらい。）

・必須項目や任意項目のタイトルが目立ち、肝心のフォーム項目は入力しづらい。

写真

・アップロード条件「3ヶ月以内の撮影、笑顔がベスト」が分かりづらい。

制約

・App Storeでは、写真登録を必須にしてはいけないが、必須になっている → ただ、大切な情報のため出来るだけ写真を登録してもらう必要がある。

解決

全体として直感的に判別、入力しやすいフォームに変更しました。

全体

・タイトルは記載せず、必須情報のみを各フォーム横に配置。フォーム自体も入力しやすいスタイルに変更。

写真

・アップロード条件「3ヶ月以内の撮影、笑顔がベスト」がすぐに目に入ってくるよう写真横に配置。その際に右利きが多いユーザーを想定して写真部分は右にした。

その他、制約

・App Storeの規約上では写真登録を必須にはできないが、店舗にとって重要な情報でもあるため、モーダルにて登録促進を促すことに。

・姓名とふりがなには「姓と名の間にスペースを入れる」必要があったため、それが直感的に分かるよう入力と同時に伝える工夫をした。

店舗 募集案件管理

課題

以前は、募集管理詳細画面で募集の編集＋選考管理ができるようになっており、 一画面において募集、選考管理のどちらがメインなのか分からずじまいで混乱を招いていました。 募集案件の削除や複製をするためのタブのナビゲーションが一見応募者の削除、複製であるかのように思えます。

流れ

募集管理トップ → 募集管理詳細（募集編集 + 選考管理）

解決

そこで、あくまでも募集管理をメインにおき、募集詳細画面に紐づく情報として選考管理を別画面で設け、遷移させることにしました。

流れ

募集管理トップ → 募集詳細 → 選考管理

本当は、管理頻度も高い応募者管理メインの画面を用意し、タブにもう一つメニューを設けたかったところですが、今回は大きな設計は変更しない前提でしたので、上記の変更にとどめました。

Design System

複数のフリーランスデザイナーさん達と取り組んでいただのですが、スピード重視なこと、 しっかりとしたデザインシステムがなくともある程度データを見て各々進めることができたため、現段階では最低限のコンポーネント整理と スタイルの定義にとどめました。
